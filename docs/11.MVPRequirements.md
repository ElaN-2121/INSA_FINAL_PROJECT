# EthioCred MVP Requirements

**Document Version:** 1.0

# Purpose

This document defines the minimum viable product (MVP) requirements for the EthioCred platform.

It serves as the project's **Definition of Done**, ensuring that all essential features are implemented before the MVP is considered complete.

All development decisions should align with the system architecture and documentation contained within the `docs/` directory.

# Project Goal

Develop a secure academic credential verification platform that enables universities to issue digitally signed academic credentials, students to manage them through a personal wallet, and employers to verify their authenticity using cryptographic verification.

The MVP should demonstrate:

- Secure authentication
- Digital credential issuance
- Cryptographic verification
- Credential revocation
- Protection against common forgery attacks

Future blockchain integration is documented but **not implemented** in the MVP.

# Functional Requirements

## Authentication

The system must support:

- Administrator login
- Registrar login
- Student login
- Employer login
- JWT authentication
- Role-Based Access Control (RBAC)
- Secure password hashing using bcrypt

## University Portal

The University Portal must allow authorized registrars to:

- Log in securely
- Upload graduation CSV files
- Validate uploaded data
- Display a staging dashboard
- Issue credential batches
- View issuance history
- Revoke previously issued credentials

## User Wallet

The User Wallet must allow students to:

- Log in using their Fayda ID (mock implementation)
- View issued credentials
- View credential details
- Receive employer verification requests
- Approve verification requests
- Reject verification requests

## Employer Portal

The Employer Portal must allow employers to:

- Log in securely
- Submit verification requests
- Search credentials
- View verification status
- Display verification results
- Scan or verify QR-code-linked credentials (optional for MVP if time permits)

# Backend Requirements

The backend must provide:

- RESTful API
- Layered architecture
- Authentication middleware
- Authorization middleware
- Validation middleware
- Service layer
- Repository layer
- Crypto service
- Audit logging
- Standardized JSON responses

# Database Requirements

Implement PostgreSQL with the required schema, including:

- Institutions
- Credentials
- Verification Requests
- Users
- Audit Logs (recommended)

Requirements:

- UUID primary keys
- Foreign key constraints
- Parameterized queries
- Indexed lookup fields

# Cryptography Requirements

The MVP must implement:

- SHA-256 hashing
- RSA-2048 digital signatures
- AES-256-GCM encryption for private key storage
- Credential canonicalization
- Digital signature verification
- Secure key loading from environment variables

The system must never expose private keys outside the backend.

# Verification Engine

The backend must verify:

- Trusted institution
- Credential integrity
- RSA signature validity
- Revocation status

A credential is considered valid only if all verification checks succeed.

# Security Requirements

The platform must implement:

- HTTPS-ready architecture
- JWT authentication
- Role-Based Access Control
- Input validation
- Secure password hashing
- Rate limiting
- CORS
- Helmet middleware
- Secure environment variables
- Audit logging

# Required Demonstrations

The MVP must successfully demonstrate the following attack scenarios:

## 1. Credential Tampering Attack

Modify credential data after issuance.

Expected result:

- SHA-256 hash changes
- RSA verification fails
- Credential rejected

## 2. Rogue Issuer Attack

Create a credential signed by an untrusted institution.

Expected result:

- Institution not found in Trust Registry
- Verification rejected

## 3. Revoked Credential Attack

Verify a credential that has been revoked.

Expected result:

- Signature remains valid
- Revocation check fails
- Credential rejected

# Out of Scope (Not Required for MVP)

The following features are intentionally excluded from the MVP but planned for future versions:

- Hyperledger Fabric integration
- Blockchain-based credential storage
- W3C Verifiable Credentials (VC)
- Decentralized Identifiers (DID)
- Mobile application
- Hardware Security Modules (HSM)
- AI-assisted fraud detection
- Multi-language support
- Real-time notifications

# Definition of Done

The EthioCred MVP is complete when:

1. A university can issue digitally signed academic credentials.
2. Students can securely access their credentials through the User Wallet.
3. Employers can verify credential authenticity using cryptographic verification.
4. Revoked credentials are rejected.
5. Forged credentials are detected.
6. Unauthorized issuers are rejected.
7. All APIs, frontend applications, and database components operate together successfully.
8. The system passes the planned security demonstrations.

Only after all requirements in this document have been satisfied should the project be considered a completed MVP.
